## Directory Files

-   Letter "d" at beginning identifies them as directory.

### Block and Character (raw) Special Device Files

-   all hardware has device file in /dev
-   Used by system to communicate with device
-   "c" or "b" in ls listing
-   Each device driver is assigned a unique number called the major number

**Major Number**

-  Kernel uses to recognize device driver type
* Major 8 represents the block device driver for SATA disks
* Major 253 denotes the driver for device mapper (VDO volumes, LVM logical volumes, etc)
* Major 11 signifies optical devices

**Minor Number**

-   applies to disk partitions as well
-   Each device controlled by the same Device Driver gets a Minor Number
-   The same driver can control multiple devices of the same type

[vagrant@server1 ~]$ ls -l /dev/sda

brw-rw—-. 1 root disk 8, 0 Dec 29 12:04 /dev/sda

^ column 5 and 6 represent the major and minor numbers

## Archiving tools

* Preserves general file attributes such as ownership, owning group, and timestamp
* preserves extended file attributes such as ACLs and SELinux contexts
* syntax of tar and star are identical

### star

### tar (tape archive)

* create, append, update, list, and extract files or an entire directory tree to and from a single file called a tarball or tarfile
* can also compress tarball after it's been created

**-c** creates tarball
-**f** specifies tarball name
**-p** Preserve file permissions. Default for the root user. Specify this if you create an archive as a normal user
**-r** appends files to the end of an extant uncompressed tarball
**-t** List contents of a tarball
**-u** Appends files to the end of an extant uncompressed tarball provided the specified files being added are newer
**-v** verbose
**-x** Extracts or restores from a tarball
**-P** override the removing of "/"
**-j** Compress tarball with bzip2
**-z** Compress tarball with gzip
-C Change to DIR before performing any options that follow

### Archive entire home directory:

tar -cvf /tmp/home.tar /home

- Removes "/" so you do not have to specify the full pathname  when restoring files at any location

### Archive two specific files:

[vagrant@server1 ~]$ tar -cvf /tmp/files.tar /etc/passwd /etc/yum.conf

### Append files in a directory to existing tarball:

[vagrant@server1 ~]$ tar -rvf /tmp/files.tar /etc/yum.repos.d

### List what is included in home.tar tarball

[vagrant@server1 ~]$ tar -tvf /tmp/files.tar

### restore single file and confirm

[vagrant@server1 ~]$ tar -xf /tmp/files.tar etc/yum.conf

[vagrant@server1 ~]$ ls -l etc/yum.conf

### Restore all files and confirm

# File Linking

### inode (index node)
A file’s metadata (128 bytes) includes: 
- file type
- size
- permissions
- owner’s name
 -owning group name 
 -last access/modification times
 -link count
 -number of allocated blocks
 -pointers to the data storage location.  This tiny storage space is referred to as the file’s 

- assigned a unique numeric identifier that is used by the kernel for accessing, tracking, and managing the file.
- mapping between an inode and a filename is referred to as a link.
- does not store the filename in its metadata
- filename and corresponding inode number mapping is maintained in the directory’s metadata where the file resides.
- Links are not created between files and directories

### Hard links

- mapping between one or more filenames and an inode number, making all hard-linked files indistinguishable from one another.
- all hard-linked files will have identical metadata.
- Changes to the file metadata and content can be made by accessing any of the filenames.
- cannot cross file system boundaries
- cannot link directories

# Soft Links

- symbolic or sumlink
- like a microsoft shortcut
- Each soft link has unique inode number
- link count does not increae or decrease
- Size of soft link is the number of character in pathname to target
- can cross file system boundaries 
- can link directories

![250](Pasted%20image%2020230131140523.png)

ln -s file10 soft10

- ls-l shows l at the beginning of the permissions for soft link
- if you remove the original file, the softlink will point to a file that doesn't exist
- RHEL 8 has four soft-linked directories under /.
	1. bin -> usr/bin
	2. lib -> usr/lib
	3. lib64 ->usr/lib64
	4. sbin -> usr/sbin
- Same syntax for creating linked directories

### Copying vs linking

copying
- duplicates source file
- each copy stores data at a unique location
- each copied file has a unique inode number and unique metadata
- If a copy is moved, erased, or renamed, the source file will have no impact, and vice versa.
- Copy is used when the data needs to be edited independent of the other.
- Permissions on the source and the copy are managed independent of each other.

Linking
- Creates a shortcut that points to the source file. The source can be accessed or modified using either the source file or the link.
- All linked files point to the same data.
- Hard Link: All hard-linked files share the same inode number, and hence the metadata. 
- Symlink: Each symlinked file has a unique inode number, but the inode number stores only the pathname to the source.
- Hard Link: If the hard link is weeded out, the other file and the data will remain untouched. 
- Symlink: If the source is deleted, the soft link will be broken and become meaningless. If the soft link is removed, the source will have no impact.
- Links are used when access to the same source is required from multiple locations.
- Permissions are managed on the source file.
